# 菜谱修改/编辑功能 — 技术路线分析

## 现状

当前项目是纯前端 + JSON 文件存储，菜谱数据通过 `recipes_full.json` 加载，编辑菜谱需要手动修改 JSON 文件。

---

## 路线一：弹窗内就地编辑（推荐起步）

在菜谱详情弹窗中，每个字段旁加一个 ✏️ 编辑按钮，点击后该字段变为可编辑状态（input/textarea），修改后保存到内存 + localStorage。

### 实现细节

- 字段切换：点击编辑 → 文字变 input → 失焦/回车保存
- 存储：修改后的菜谱存到 `localStorage`（`editedRecipes` 对象），加载时优先读取 localStorage 覆盖原始数据
- 导出：提供"导出修改后的 JSON"按钮，下载修改后的 `recipes_full.json`

### 优点
- 交互直观，原地编辑，不需要跳转
- 不需要后端，完全纯前端
- 实现简单，1-2天可完成

### 缺点
- 无法添加新字段（如新增食材种类）
- 不能新增/删除菜谱
- 刷新页面后修改仍在（localStorage），但清除缓存会丢失

---

## 路线二：独立编辑面板

在筛选栏增加一个「✏️ 编辑模式」按钮，进入编辑模式后：
- 卡片上出现编辑/删除按钮
- 点击编辑弹出一个独立编辑表单（所有字段可编辑）
- 支持新增菜谱、删除菜谱

### 存储方案
- localStorage 存完整菜谱库（初始加载时从 JSON 复制到 localStorage）
- 每次修改后更新 localStorage
- 用户可导出完整 JSON 文件

### 优点
- 完整 CRUD 支持
- 不依赖后端
- 用户体验完整

### 缺点
- 多设备不同步
- 数据存储在浏览器，可能丢失
- 实现复杂度中等，3-5天

---

## 路线三：GitHub-based 编辑（推荐长期方案）

利用 GitHub API 直接编辑仓库中的 `recipes_full.json`：

### 实现流程
1. 用户点击编辑 → 弹出编辑表单
2. 修改完成 → 通过 GitHub API 提交 PR 或直接 push 到仓库
3. Vercel 自动重新部署

### 技术栈
- GitHub OAuth App 做用户认证
- GitHub REST API（`PUT /repos/:owner/:repo/contents/:path`）
- 或者用 Netlify CMS / Decap CMS 集成

### 优点
- 数据持久化，多设备同步
- 天然版本控制（Git 历史）
- 可以配置 PR 审核流程

### 缺点
- 需要 GitHub 账号
- 实现复杂度高，需要 OAuth
- 5-10天开发

---

## 路线四：后端化 + 数据库

引入轻量后端（如 Supabase）：

### 实现
- 菜谱数据迁移到 Supabase PostgreSQL
- 前端通过 Supabase JS SDK 读写
- 用户系统（可选）：登录后收藏、编辑

### 优点
- 真正的数据持久化
- 可扩展（用户系统、评论、评分）
- Supabase 免费额度充足

### 缺点
- 需要迁移现有数据
- 依赖外部服务
- 需要重构前端数据加载逻辑

---

## 路线五：使用 parse_recipe.py 变体（文本编辑）

当前已有 `parse_recipe.py` 脚本，可以做一个"文本编辑器"：

### 实现
- 菜谱详情弹窗中加一个「📝 编辑源码」按钮
- 点击后显示该菜谱的 Markdown/文本格式
- 用户修改文本后，调用 parse_recipe.py 的逻辑（前端 JS 实现）重新解析
- 把解析结果更新到 localStorage

### 优点
- 复用现有解析逻辑
- 适合习惯用文本编辑的用户
- 实现简单

### 缺点
- 文本格式可能丢失
- 解析可能不完美

---

## 推荐组合方案

```
Phase 1（立即）：路线一（弹窗就地编辑）+ localStorage
  → 支持修改标题、食材、步骤、技巧等基本字段

Phase 2（1-2周）：路线二（独立编辑面板）
  → 完整 CRUD，支持新增/删除菜谱

Phase 3（长期）：路线三（GitHub API）或路线四（Supabase）
  → 数据持久化、多设备同步
```

| 路线 | 开发量 | 持久化 | 同步 | 推荐场景 |
|------|--------|--------|------|----------|
| 一：弹窗编辑 | 1-2天 | localStorage | ❌ | 快速起步 |
| 二：编辑面板 | 3-5天 | localStorage | ❌ | 完整CRUD |
| 三：GitHub API | 5-10天 | ✅ Git | ✅ | 开源协作 |
| 四：Supabase | 5-7天 | ✅ DB | ✅ | 用户系统 |
| 五：文本编辑 | 1天 | localStorage | ❌ | 极客向 |
